# Analysis Agent 개발 내역 정리

## 프로젝트 개요

영양소 분석 및 영양제 추천 Agent.
AWS Bedrock AgentCore Runtime에 컨테이너로 배포되며,
여러 MSA 서비스에서 공통으로 호출하는 구조.

---

## 아키텍처 최종 확정

```
App (다른 MSA 서비스들)
  └─► invoke_agent_runtime()
        └─► AgentCore Runtime (ECR 컨테이너)
              ├─ Step 1: Bedrock LLM 직접 호출
              │           → 건강검진 + 복용영양제 + 섭취목적 분석
              │           → required_nutrients 도출
              │
              ├─ Step 2: Lambda 호출 (action-nutrient-calc)
              │           → 갭 계산 (DB 접근 없음, 순수 계산)
              │           → current_supplements, unit_cache는
              │             App이 DB에서 조회해서 payload로 전달
              │
              └─ Step 3: Bedrock LLM 직접 호출
                          → gaps 기반 영양제 추천
                          → recommendations 생성
              │
              최종 JSON 반환 { step1, step2, step3 }
                │
                App이 받아서 RDS에 저장
                ├─ step1 → analysis_result
                ├─ step2 → nutrient_gap
                └─ step3 → recommendations
```

---

## 아키텍처 결정 과정

### Lambda vs FastAPI 내부 함수 논의

**팀 초기 계획**: LLM이 수학 계산을 못하니까 Lambda 사용

**결론**: RDS 사용 환경에서 Lambda의 단점이 명확해서 방향 수정

| 항목 | Lambda | FastAPI 내부 |
|---|---|---|
| RDS 접근 | VPC 설정 필요 | 커넥션 풀 재사용 |
| Cold Start | VPC 내 1~3초 | 없음 |
| RDS Proxy | 필요 ($20~30/월) | 불필요 |
| Terraform 리소스 | 증가 | 없음 |

**단, Step 2 갭 계산은 Lambda 유지** — 이유:
- Agent가 1번 호출로 3단계를 처리하려면 Lambda가 필요
- Lambda가 DB 저장을 안 하고 계산만 하면 VPC 설정 불필요
- `current_supplements`, `unit_cache`를 App이 payload로 전달 → Lambda는 순수 계산만

### DB 저장 주체

**결론**: App이 AgentCore 응답 받은 후 RDS에 직접 저장

- AgentCore Runtime은 추론만 담당
- DB 저장 로직은 App 서비스에 `AgentCoreClient` 클래스로 구현
- 여러 서비스에서 재사용 가능

### AgentCore Runtime 선택 이유

- Agent를 독립 서비스로 배포 → 여러 MSA에서 공통 호출
- ECR 이미지로 패키징 → GitOps CI/CD 연동 가능
- IAM Role 기반 인증 → Access Key 불필요

## DB 스키마 변경 사항

기존 스키마 대비 수정 내용:

```sql
-- 1. 테이블명 오타 수정
anaysis_current_ingredients → analysis_current_ingredients

-- 2. 수치 컬럼 타입 변경 (단위 변환 소수점 보존)
nutrient_gap.current_amount    INTEGER → NUMERIC(10,4)
nutrient_gap.gap_amount        INTEGER → NUMERIC(10,4)
product_nutrients.amount_per_serving  INTEGER → NUMERIC(10,4)
product_nutrients.amount_per_day      INTEGER → NUMERIC(10,4)
analysis_current_ingredients.ans_nutrient_amount INTEGER → NUMERIC(10,4)

-- 3. recommendations FK 추가 (기존 스키마 누락)
recommendations.result_id  → analysis_result.result_id
recommendations.product_id → products.product_id
```

### IAM Role (CLI로 임시 생성 — 추후 Terraform으로 이관 필요)

| Role | 용도 |
|---|---|
| `lambda-basic-role` | Lambda 실행 Role |
| `agentcore-runtime-role` | AgentCore Runtime 실행 Role |

### agentcore-runtime-role 권한

- `AmazonBedrockFullAccess` — Bedrock 모델 호출
- `ecr-pull` (인라인) — ECR 이미지 풀
- `invoke-lambda` (인라인) — action-nutrient-calc Lambda 호출

## 로컬 배포 순서 (수동)

```bash
# 1. 테스트
pytest tests/ -v

# 2. Docker 빌드 (arm64)
docker build --platform linux/arm64 \
  -t 070238434919.dkr.ecr.ap-northeast-2.amazonaws.com/analysis-agent:버전 \
  .

# 3. ECR 로그인 & 푸시
aws ecr get-login-password --region ap-northeast-2 | \
  docker login --username AWS --password-stdin \
  070238434919.dkr.ecr.ap-northeast-2.amazonaws.com

docker push 070238434919.dkr.ecr.ap-northeast-2.amazonaws.com/analysis-agent:버전

# 4. Lambda 업데이트
cd lambdas/action_nutrient_calc
zip function.zip handler.py
aws lambda update-function-code \
  --function-name action-nutrient-calc \
  --zip-file fileb://function.zip

# 5. AgentCore Runtime 업데이트
aws bedrock-agentcore-control update-agent-runtime \
  --agent-runtime-id analysis_agent-bA2YOVDhUj \
  --agent-runtime-artifact '{"containerConfiguration": {"containerUri": "070238434919.dkr.ecr.ap-northeast-2.amazonaws.com/analysis-agent:버전"}}' \
  --role-arn arn:aws:iam::070238434919:role/agentcore-runtime-role \
  --region ap-northeast-2
```
